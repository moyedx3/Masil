## Judging Criteria (100점)

### 1) Problem & Human-Only Fit (30점)
- 해결하려는 문제가 명확하고 실제로 존재하는가?
- World ID / Proof of Personhood가 핵심 전제인가?
- World ID가 없으면 서비스가 성립하지 않거나 명확히 약해지는가?

### 2) Privacy-by-Design & Trust (20점)
- 데이터 최소화(Yes/No, 속성 증명) 구조인가?
- ZKP / VC를 개념적으로 올바르게 이해하고 적용했는가?
- 사용자가 프라이버시·신뢰를 직관적으로 이해할 수 있는가?

### 3) Technical Execution & Anti-Abuse (20점)
- 실제 동작하는 데모/프로토타입 또는 설득력 있는 플로우가 있는가?
- 재사용·리플레이·대리 인증 등 어뷰징 리스크를 인지하고 있는가?
- 기술 선택이 과하지 않고 핵심에 집중했는가?

### 4) Product Craft & UX Flow (15점)
- 온보딩 → 인증 → 액션까지 흐름이 자연스러운가?
- 사용자가 "뭘 했고, 왜 안전한지"를 3초 안에 이해하는가?
- 미니앱/웹/오프라인 연계 UX가 깔끔한가?

### 5) Pitch, Feasibility & Impact (15점)
- 2–3분 안에 문제 → 해결 → 왜 World ID → 데모 → 다음 단계가 전달되는가?
- 실제 사용처(상점·이벤트·커뮤니티·DAO 등)가 그려지는가?

## 프로젝트 발표 안내

### 발표 형식
- **발표:** 5분
- **Q&A:** 2분
- **발표 언어:** 자유 (슬라이드는 영어 권장)
- **발표자 수:** 제한 없음 (시간 엄수)

⛔️ 시간 초과 시 감점될 수 있음

### 슬라이드 구성 (최대 6장)

#### 1️⃣ 문제 (1문장)
- 지금 존재하는 명확한 문제 하나
- 실제 사용자/현장에서 바로 이해되는 표현

#### 2️⃣ 솔루션 (1문장 + 그림)
- 당신의 미니앱/서비스가 무엇을 해결하는지
- 다이어그램, 플로우, 화면 목업 권장

#### 3️⃣ 왜 Human-only / World ID 인가
- World ID가 없으면 왜 성립하지 않는지
- 단순 로그인/편의성 설명 ❌
- "봇·다중계정·대리행위 방지" 등 핵심 이유 명시

#### 4️⃣ 데모
아래 중 하나 이상 필수:
- GIF
- 화면 녹화 영상
- 클릭 가능한 목업(Figma 등)

선택: 라이브 데모

👉 완벽한 구현보다 "흐름이 이해되는 데모"가 중요

#### 5️⃣ 프라이버시 & 리스크 대응
- Yes / No 속성 증명 여부
- 어떤 데이터를 수집하지 않는지
- 재사용·대리인증·남용에 대한 기본 대응

(예: "생년월일 저장 ❌ / 성인 여부 Yes/No만")

#### 6️⃣ 향후 계획 (Go-to-Market 포함)
- 해커톤 런칭 후 무엇을 만들 것인가
- 실제 사용 시나리오 (상점, 이벤트, 커뮤니티, DAO 등)
- 간단한 배포/확장 계획

### 중요한 기준
- World ID는 '옵션'이 아니라 '핵심 전제'여야 함
- 기술 복잡도보다 문제 적합성 + 설득력
- "우리는 왜 이걸 지금 만들어야 하는가?"에 답해야 함

### 좋은 발표의 기준 (요약)
3분 안에 심사위원이 아래를 이해하면 성공:
- 어떤 문제인가
- 왜 Human-only인가
- 실제로 어떻게 쓰이는가
- 지금 당장 만들 수 있는가
